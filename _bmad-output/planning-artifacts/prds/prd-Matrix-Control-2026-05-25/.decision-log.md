## How to read this log

Each entry records a brownfield decision for an existing code zone:

| Decision | Meaning |
|---|---|
| **KEEP** | Keep as-is — wire/complete only, no refactor |
| **KEEP + IMPROVE** | Keep architecture, targeted improvements (tests, perf, polish) |
| **REWIRE** | Keep UI/code, replace stub wiring (e.g. timestamps → Core) |
| **REFACTOR** | Partial or full rework of the zone |
| **DISCARD** | Abandon (e.g. OpenGL plans, temporary dev patterns) |
| **BUILD NEW** | Does not exist yet — specify from scratch |

## Phase 0 decisions

### Zone 1 — Descriptors / APVTS / WidgetFactory / Validators

#### D-001 — Descriptor-driven architecture (PluginIDs, PluginDisplayNames, PluginDescriptors)

- **Decision:** KEEP + IMPROVE
- **Rationale:** Guillaume confirme que les grands principes (source de vérité unique, hiérarchie Matrix-1000, ParameterWidgets vs StandaloneWidgets, plan de construction → factories) sont un souhait fondateur et restent l'invariant v1. Le code actuel (partiellement vibecodé) reflète cette intention ; améliorations attendues sur découpage fichiers, lisibilité « livre ouvert », et documentation — pas sur le modèle conceptuel.
- **Follow-up:** Étape Architecture BMad + polish ciblé des sources ; pas de refonte du modèle descriptor-driven.

#### D-002 — ApvtsFactory / ApvtsLayoutBuilder / WidgetFactory

- **Decision:** KEEP + IMPROVE
- **Rationale:** Chaîne APVTS ← descriptors → widgets alignée avec la vision. Factories = « artisans » du plan ; à rendre plus lisibles pour un dev humain reprenant le projet dans 6–12 mois.
- **Follow-up:** Revue structurelle lors de `bmad-create-architecture` ; critère acceptation = un contributeur comprend le flux sans l'auteur original.

#### D-003 — Validateurs au boot (ApvtsValidator, WidgetFactoryValidator)

- **Decision:** KEEP + IMPROVE
- **Rationale:** Validation au lancement souhaitée (« pas de trou dans la raquette »). Aujourd'hui : ApvtsValidator → `jassertfalse` debug only ; WidgetFactoryValidator → messages footer via ExceptionPropagator (non bloquant). Améliorer UX en release si incohérence détectée, sans frustration injustifiée.
- **Follow-up:** PRD NFR ou addendum : politique dev (assert) vs prod (message diagnostic + mode dégradé si pertinent).

#### D-004 — Patch Manager et APVTS

- **Decision:** KEEP (objet APVTS unique) + REWIRE (clarifier le modèle)
- **Rationale:** Ce n'est pas « hors APVTS » — le Patch Manager vit dans `apvts.state` via **ValueTree properties**, pas via `AudioParameter*`. Choix intentionnel ou semi-intentionnel de l'IA : pas de paramètres automatisables DAW pour banque/patch/boutons manager. Guillaume veut un seul canal Processor↔Editor : **APVTS** (params + state properties). Aligné avec JUCE.
- **Follow-up:** Documenter la règle « ParameterWidget = AudioParameter ; StandaloneWidget / état PM = property sur apvts.state ». Envisager descripteurs de properties PM dans PluginDescriptors pour fermer les trous validateur. Ne pas migrer tout le PM en AudioParameters sauf besoin automation explicite.

#### D-005 — StandaloneWidgets (clics, toggles, état UI)

- **Decision:** KEEP (pattern properties) + REWIRE (sémantique)
- **Rationale:** Boutons I/C/P et PM : `setProperty(id, timestamp)` — pattern clic valide ; toggles Mutator : bool. `InternalPatchesPanel` écoute bank/patch → NumberBox (Core→GUI partiel). Pas de mécanisme général Core→GUI pour enable/disable boutons standalone aujourd'hui. Timestamps = stub en attente d'ActionDispatcher.
- **Follow-up:** Zone 5 workshop + BUILD NEW ActionDispatcher ; convention property documentée (event timestamp vs état persistant vs enabled flag).

#### D-006 — Matrix-6/6R (v1 PATCH only)

- **Decision:** KEEP
- **Rationale:** Même hardware PATCH et SysEx que M-1000 ; pas de descripteurs/limites séparés. MASTER grisé ; SPLIT ignoré.
- **Follow-up:** FR PRD : comportement UI device-type ; pas d'extension descriptor PATCH.

#### D-007 — ScaleLevels dans PluginIDs.h

- **Decision:** KEEP + IMPROVE
- **Rationale:** Ce ne sont pas des string IDs mais des constantes métier (IDs combo 1–7 + table float). Placement discutable ; commentaire header déjà partiel. Déplacer vers module Settings/UiScale dédié ou documenter l'exception.
- **Follow-up:** Refactor mineur lors du polish Architecture — hors chemin critique v1.

#### D-008 — sysExId = kNoSysExId (MASTER)

- **Decision:** KEEP + IMPROVE
- **Rationale:** Sémantique correcte (MASTER SysEx diffère). Expression peut être affinée (type optionnel, descripteur MasterParameter) sans changer le modèle.
- **Follow-up:** Suggestion refonte légère des types descripteurs — Architecture, pas bloquant PRD.

#### D-009 — Documentation descriptor stack + stratégie doc JUCE

### Zone 3 — Afficheurs interactifs (EnvelopeDisplay, TrackGeneratorDisplay)

#### D-016 — Comportement interactif v1 (slider ↔ courbe + souris)

- **Decision:** KEEP
- **Rationale:** v1 exige les deux sens : slider met à jour la courbe ; clic-glisser sur la courbe met à jour le ou les sliders concernés. Déjà implémenté ; comportement validé et satisfaisant pour EnvelopeDisplay et TrackGeneratorDisplay.
- **Follow-up:** FR PRD : bidirectionnel + édition souris obligatoires v1.

#### D-017 — Marqueurs visuels sur les courbes (ronds pleins/creux → carrés)

- **Decision:** KEEP + IMPROVE
- **Rationale:** Distinction ronds pleins/creux (sens de déplacement) jugée inutile et peu esthétique ; TrackGeneratorDisplay incohérent (tous creux). Remplacer par **petits carrés pleins** partout — cohérent avec look GUI 80's anguleux (Matrix-1000). Simplification volontaire : plus de sémantique plein/creux.
- **Follow-up:** Implémentation widgets ; mockup/Figma si besoin ; homogénéiser ENV ×3 + Track.

#### D-018 — Architecture sync Display / Slider / APVTS

- **Decision:** REFACTOR — **implémenté 2026-05-27**
- **Rationale:** Chemin actuel Display → Slider → APVTS = détour fragile (Guillaume ne se souvient plus du choix initial). Cible : **APVTS seule source de vérité** pour les valeurs 0–63. Souris sur courbe → écriture directe param APVTS ; attachments JUCE mettent à jour les sliders en retour. Supprimer le pont manuel Display→setValue(slider) dans PatchEditPanel. Refactorer PatchEditDisplaysPanel (mapping paramId→display, fin de la longue chaîne if). PluginProcessor intervient pour effets métier (SysEx, rate limiting) — pas pour resynchroniser sliders. Pattern standard plugins JUCE.
- **Follow-up:** `InteractiveDisplayApvtsSync` (nouveau) ; retirer ~150 lignes pont slider dans `PatchEditPanel` ; gestures souris via `beginChangeGesture`/`endChangeGesture`. Tests manuels slider↔courbe + automation DAW.

#### D-019 — Géométrie et mapping visuel des courbes (0–63)

- **Decision:** KEEP (règles actuelles) + KEEP + IMPROVE (validation terrain)
- **Rationale:** Plage [0 ; 63] = calibrage SysEx Matrix-1000 (6 bits), pas une approximation arbitraire. Règles détaillées enveloppe (Delay/Attack/Decay/Sustain/Release : proportions ~20 % largeur par segment temporel, sustain horizontal translatable verticalement, Decay non draggable en Y direct pour éviter double SysEx, segments non verticaux à 0 pour esthétique) et Track (5 points espacés ~25 % largeur, déplacement Y only) = intention produit validée. Clarification PM : la phrase « pas une photo du synthé » visait le **rendu graphique** (interprétation à l'écran), pas la plage de valeurs — pas de changement de calibrage proposé.
- **Follow-up:** Retest conditions réelles avec synthé connecté ; specs géométrie dans addendum.md ; FR PRD renvoie aux règles de mapping.

#### D-020 — Perf rendu et UI Scale (150–200 %)

- **Decision:** KEEP (v1)
- **Rationale:** Pas de ralentissement/saccade observé après travaux UI Scale ; suppression cache image = plus fluide qu'avec cache. Suffisant pour v1 si retests terrain OK.
- **Follow-up:** Tests réels pre-launch ; améliorations archi/perf seulement si retest ou Architecture identifie un gain — non bloquant v1.
