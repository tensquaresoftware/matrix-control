## How to read this log

Each entry records a brownfield decision for an existing code zone:

| Decision | Meaning |
|---|---|
| **KEEP** | Keep as-is — wire/complete only, no refactor |
| **KEEP + IMPROVE** | Keep architecture, targeted improvements (tests, perf, polish) |
| **REWIRE** | Keep UI/code, replace stub wiring (e.g. timestamps → Core) |
| **REFACTOR** | Partial or full rework of the zone |
| **DISCARD** | Abandon (e.g. OpenGL plans, temporary dev patterns) |
| **BUILD NEW** | Does not exist yet — specify from scratch |

- **Decision:** KEEP + IMPROVE

#### D-002 — ApvtsFactory / ApvtsLayoutBuilder / WidgetFactory

- **Decision:** KEEP + IMPROVE
- **Rationale:** Chaîne APVTS ← descriptors → widgets alignée avec la vision. Factories = « artisans » du plan ; à rendre plus lisibles pour un dev humain reprenant le projet dans 6–12 mois.

### Zone 7 — MidiManager + thread MIDI dédié

#### D-048 — API publique MidiManager (impérative + queue interne)

- **Decision:** KEEP (API impérative publique) + BUILD NEW (queue interne)
- **Rationale:** Les appelants (ActionDispatcher, PluginProcessor, PatchFileService) expriment une **intention métier** (`sendPatch`, `requestCurrentPatch`) — pas un détail d'implémentation queue. En interne, chaque appel **enqueue** ; seul le thread MIDI consomme et parle au hardware. Avantages queue : pas de blocage UI/audio sur I/O MIDI ; rate-limit centralisé ; priorité notes/CC > SysEx ; testabilité (ordre, débit) ; protection Matrix-1000 contre flood SysEx (critère #1 brief). L'API publique reste lisible « livre ouvert » ; la queue est un détail d'implémentation du thread dédié.
- **Follow-up:** Architecture diagramme producteurs → `MidiOutboundQueue` → `MidiManager::run()` → `MidiSender` ; tests unitaires queue (D-047).

#### D-049 — Nommage méthodes MidiManager (send vs request)

- **Decision:** KEEP + IMPROVE — convention **`sendPatch` / `sendMaster` / `requestCurrentPatch` / `requestMaster`**
- **Rationale:** Verbe = direction du flux : **send** = plugin → synthé ; **request** = plugin demande une réponse SysEx. Nom domaine sans suffixe `Data` redondant — « Patch » et « Master » suffisent dans le contexte Matrix-1000. Renommer `requestMasterData` → `requestMaster` pour parité avec `requestCurrentPatch`. Rejeter `sendPatchData` / `sendMasterData` / `*Data` partout : bruit sans gain sémantique. Rejeter aussi `sendMasterData` seul : asymétrie arbitraire.
- **Follow-up:** Refactor rename au câblage v1 ; grep codebase + tests.

#### D-050 — Modèle thread + inbound (sync vs async)

- **Decision:** KEEP + IMPROVE — **hybride A** (sync ciblé + async session)
- **Rationale:** **Sync (request/response bloquant)** pour opérations explicites à issue connue : Device Inquiry au startup, `requestCurrentPatch` / `requestMaster` après action utilisateur (STORE, chargement banque). L'utilisateur accepte une attente courte ; UI déjà verrouillée sans device (D-010). **Async** pour le reste : `MidiManager::run()` consomme la queue outbound ; callback inbound SysEx (thread JUCE MIDI input) notifie le thread MIDI via file/flag — pas de decode APVTS dans le callback. Mises à jour APVTS depuis inbound : `MessageManager::callAsync()` (jamais depuis callback MIDI ni audio). `waitForSysExResponse` conservé pour le chemin RPC ; `handleIncomingSysEx` devient le routeur async (réponses spontanées ou complétion de requêtes en attente). Rejeter **B** (tout sync sur thread MIDI) : bloque la queue pendant les timeouts. Matrix-1000 envoie surtout des **réponses** à nos requêtes — le modèle RPC couvre l'essentiel v1 ; chemin async prêt pour inbound non sollicité.
- **Follow-up:** Architecture séquence startup (inquiry → unlock UI) ; documenter quels appels sont sync vs fire-and-forget.

#### D-051 — Queue unifiée outbound (dual-role)

- **Decision:** BUILD NEW — `MidiOutboundQueue` (ou équivalent)
- **Rationale:** Brief non négociable : une sortie **MIDI To**, deux producteurs. **Instrument** (`processBlock` / Keyboard From standalone) : notes, CC, pitch bend — **enqueue only**, filtrer PC et SysEx. **Editor** (thread message / ActionDispatcher) : SysEx, Program Change — enqueue. Consommateur unique : thread MIDI — dequeue avec **priorité realtime > SysEx**, délai minimum entre SysEx (`kMinSysExDelayMs`, timings PRD). Jamais d'appel direct `MidiOutput` depuis audio thread. LEDs header Instrument/Editor alimentées à l'enqueue/dequeue (impulsion ou compteur — détail UX).
- **Follow-up:** FR PRD timings rate-limit ; tests queue sous charge automation ; zone 8 `processBlock` enqueue instrument.

#### D-052 — Granularité classes transport (Sender / Receiver / Ports)

- **Decision:** KEEP (découpage SRP) + REWIRE + BUILD NEW (queue séparée) — **pas** fusion en mega-classe
- **Rationale:** **Consolider** = regrouper la **logique queue/dispatch**, pas fusionner Sender+Receiver+Ports en un blob. `MidiInputPort` / `MidiOutputPort` = wrappers ouverture port (KEEP). `MidiSender` = écriture physique **appelée uniquement depuis thread MIDI** (KEEP, affiner). `MidiReceiver` = callback JUCE + buffer SysEx — **REWIRE** : aujourd'hui `handleIncomingSysEx` dans MidiManager n'est jamais appelé depuis le Receiver (disconnect) ; le Receiver doit notifier MidiManager (file lock-free ou callback enregistré). **BUILD NEW** : `MidiOutboundQueue` (+ éventuellement `MidiInboundDispatcher`) — classes dédiées, MidiManager orchestre. Rejeter fusion totale : violerait SRP, classe > 200 lignes, tests plus difficiles.
- **Follow-up:** Architecture diagramme composants ; fix wiring Receiver → MidiManager ; ne pas appeler SysEx decode dans Receiver.
