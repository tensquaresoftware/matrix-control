## How to read this log

Each entry records a brownfield decision for an existing code zone:

| Decision | Meaning |
|---|---|
| **KEEP** | Keep as-is — wire/complete only, no refactor |
| **KEEP + IMPROVE** | Keep architecture, targeted improvements (tests, perf, polish) |
| **REWIRE** | Keep UI/code, replace stub wiring (e.g. timestamps → Core) |
| **REFACTOR** | Partial or full rework of the zone |
| **DISCARD** | Abandon (e.g. OpenGL plans, temporary dev patterns) |
| **BUILD NEW** | Does not exist yet — specify from scratch |

- **Amends:** D-022-R2 (full lock mirror driving navigation); investigation fix #4 (`clearSyncedBankState` after UNLOCK).

### Zone 5 — I/C/P stubs & ActionDispatcher

#### D-031 — GUI → Processor action trigger (timestamp pattern)

- **Decision:** KEEP (timestamp on APVTS property) + BUILD NEW (listener + ActionDispatcher)
- **Rationale:** APVTS = single Processor↔Editor channel (D-004). Each click produces a unique value → reliable edge detection, even on rapid clicks. Direct GUI callback alternative breaks the umbilical model. Counter alternative = equivalent but less debug-readable. `PluginProcessor` (message thread) implements `ValueTree::Listener` → delegates to `ActionDispatcher::onActionPropertyChanged(id, newValue)` for “action” IDs only.
- **Follow-up:** Architecture: classify properties **event** (timestamp, not persisted on plugin state save) vs **state** (persisted bool/int) vs **uiMirror** (enabled flags Core→GUI). Exclude events from session XML.

#### D-032 — ActionDispatcher architecture

- **Decision:** BUILD NEW — single hub + specialized handlers
- **Rationale:** `ActionDispatcher` owned by `PluginProcessor`; registry `widgetId → handler`. Handlers: `ModuleActionHandler` (I/C/P + Matrix Mod init), `PatchManagerActionHandler`, `MutatorActionHandler` (stub → zone 9 engine). One hub = execution order, centralizable debounce/rate-limit, unit tests per handler. No scattered listeners in the Editor.
- **Follow-up:** `bmad-create-architecture` flow diagram; no business logic in panels beyond `setProperty`.

#### D-033 — ClipboardService type-aware + Paste feedback

- **Decision:** BUILD NEW (`ClipboardService`) + REWIRE (enabled properties)
- **Rationale:** **Module clipboard**: C always active; P active only on modules compatible with the **last** module COPY (no warning on source change — last C wins). **Patch clipboard** (Internal Patches COPY): grays **all** PATCH EDIT module P buttons; only Internal Patches PASTE remains relevant. Both modes mutually exclusive — last COPY (module or full patch) redefines mode. Core pushes `pasteEnabled` per target via APVTS properties; GUI listens and grays buttons (ModuleHeader, Internal Patches).
- **Follow-up:** PRD compatibility matrix (DCO1↔DCO2 partial, ENV strict, LFO partial); ClipboardService tests.

### Zone 7 — MidiManager + dedicated MIDI thread

#### D-048 — MidiManager public API (imperative + internal queue)

- **Decision:** KEEP (public imperative API) + BUILD NEW (internal queue)
- **Rationale:** Callers (ActionDispatcher, PluginProcessor, PatchFileService) express **business intent** (`sendPatch`, `requestCurrentPatch`) — not queue implementation detail. Internally, each call **enqueues**; only the MIDI thread consumes and talks to hardware. Queue advantages: no UI/audio blocking on MIDI I/O; centralized rate-limit; notes/CC priority over SysEx; testability (order, throughput); Matrix-1000 flood protection (brief criterion #1). Public API stays “open book” readable; queue is dedicated-thread implementation detail.
- **Follow-up:** Architecture diagram producers → `MidiOutboundQueue` → `MidiManager::run()` → `MidiSender`; queue unit tests (D-047).

#### D-049 — MidiManager method naming (send vs request)

- **Decision:** KEEP + IMPROVE — convention **`sendPatch` / `sendMaster` / `requestCurrentPatch` / `requestMaster`**
- **Rationale:** Verb = flow direction: **send** = plugin → synth; **request** = plugin asks for SysEx response. Domain name without redundant `Data` suffix — “Patch” and “Master” suffice in Matrix-1000 context. Rename `requestMasterData` → `requestMaster` for parity with `requestCurrentPatch`. Reject `sendPatchData` / `sendMasterData` / `*Data` everywhere: noise without semantic gain. Also reject `sendMasterData` alone: arbitrary asymmetry.
- **Follow-up:** Refactor rename at v1 wiring; grep codebase + tests.

#### D-050 — Thread model + inbound (sync vs async)

- **Decision:** KEEP + IMPROVE — **hybrid A** (targeted sync + async session)
- **Rationale:** **Sync (blocking request/response)** for explicit operations with known outcome: Device Inquiry at startup, `requestCurrentPatch` / `requestMaster` after user action (STORE, bank load). User accepts short wait; UI already locked without device (D-010). **Async** for the rest: `MidiManager::run()` consumes outbound queue; inbound SysEx callback (JUCE MIDI input thread) notifies MIDI thread via queue/flag — no APVTS decode in callback. APVTS updates from inbound: `MessageManager::callAsync()` (never from MIDI or audio callback). `waitForSysExResponse` kept for RPC path; `handleIncomingSysEx` becomes async router (unsolicited responses or pending request completion). Reject **B** (all sync on MIDI thread): blocks queue during timeouts. Matrix-1000 mostly sends **responses** to our requests — RPC model covers most of v1; async path ready for unsolicited inbound.
- **Follow-up:** Architecture startup sequence (inquiry → unlock UI); document which calls are sync vs fire-and-forget.

#### D-051 — Unified outbound queue (dual-role)

- **Decision:** BUILD NEW — `MidiOutboundQueue` (or equivalent)
- **Rationale:** Brief non-negotiable: one **MIDI To** output, two producers. **Instrument** (`processBlock` / Keyboard From standalone): notes, CC, pitch bend — **enqueue only**, filter PC and SysEx. **Editor** (message thread / ActionDispatcher): SysEx, Program Change — enqueue. Single consumer: MIDI thread — dequeue with **realtime priority > SysEx**, minimum delay between SysEx (`kMinSysExDelayMs`, PRD timings). Never call `MidiOutput` directly from audio thread. Header Instrument/Editor LEDs driven at enqueue/dequeue (pulse or counter — UX detail).
- **Follow-up:** PRD rate-limit timings; queue tests under automation load; zone 8 `processBlock` enqueues instrument.

#### D-052 — Transport class granularity (Sender / Receiver / Ports)

- **Decision:** KEEP (SRP split) + REWIRE + BUILD NEW (separate queue) — **no** mega-class merge
- **Rationale:** **Consolidate** = group **queue/dispatch logic**, not merge Sender+Receiver+Ports into one blob. `MidiInputPort` / `MidiOutputPort` = port open wrappers (KEEP). `MidiSender` = physical write **called only from MIDI thread** (KEEP, refine). `MidiReceiver` = JUCE callback + SysEx buffer — **REWIRE**: today `handleIncomingSysEx` in MidiManager is never called from Receiver (disconnect); Receiver must notify MidiManager (lock-free queue or registered callback). **BUILD NEW**: `MidiOutboundQueue` (+ optionally `MidiInboundDispatcher`) — dedicated classes, MidiManager orchestrates. Reject full merge: would violate SRP, class > 200 lines, harder tests.
- **Follow-up:** Architecture component diagram; fix Receiver → MidiManager wiring; do not call SysEx decode in Receiver.
